# Smart Commit

変更内容を読み取り、このプロジェクトの規約（`feat:` / `fix:` / `test:` / `refactor:` / `chore:` プレフィックス）に沿ったコミットメッセージを生成してコミットする。

## 手順

1. 以下を並行して取得する:
   - `git status` で未ステージ・ステージ済みファイルの一覧
   - `git diff HEAD` で全変更差分（ステージ済み＋未ステージ）

2. 変更内容を分析してプレフィックスを決める:

   | プレフィックス | 使う場面 |
   |---|---|
   | `feat:` | 新しい機能・シナリオ・エンドポイントの追加 |
   | `fix:` | バグ修正 |
   | `test:` | テスト・ステップ定義・フィーチャーファイルのみの変更 |
   | `refactor:` | 動作を変えないコードの整理 |
   | `chore:` | ビルド設定・依存関係・ドキュメント等の変更 |

3. コミットメッセージを日本語で生成する:
   - 1行目: `<prefix>: <変更の要約>`（72文字以内）
   - BDD のフィーチャーファイルが含まれる場合は、追加したシナリオ名を要約に入れる
   - 複数ドメインにまたがる場合は最も重要な変更にフォーカスする

4. 生成したメッセージをユーザーに提示して確認を取る。
   - 承認されたら `git add -A` でステージして `git commit -m "..."` を実行する
   - 修正を求められたらメッセージを直して再提示する
   - 「このファイルは除く」と言われたら該当ファイルを除いてステージする

## コミットメッセージ例

```
feat: ユーザー登録機能を追加（重複メールエラー含む）
fix: ログイン時に無効なJWTでも認証が通る問題を修正
test: プロフィール更新のBDDシナリオを追加
refactor: UserService のコンストラクタインジェクションに変更
chore: Cucumber 7.x に依存関係を更新
```

## 注意事項

- `.env` や認証情報を含むファイルが差分にある場合は、コミットに含めないよう警告する
- `git commit` 前に必ずユーザーの承認を得る
- ステージされていない変更がある場合は、すべてまとめてコミットするか個別にするかを確認する
